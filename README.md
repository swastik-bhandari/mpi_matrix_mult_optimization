# MPI Matrix Multiplication — Naive vs Cannon's Algorithm

Two implementations of distributed matrix multiplication using MPI, with a full performance analysis using the alpha-beta communication model.

**Manually:**
```bash
mpicxx -O3 -march=native -o src/naive_matmul src/naive_matmul.cpp
mpicxx -O3 -march=native -o src/cannon_matmul src/cannon_matmul.cpp
```

## Running

```bash
# Naive: any number of processes, n must be divisible by p
mpirun -np 4 src/naive_matmul 1024

# Cannon: p must be a perfect square (1, 4, 9, 16, ...)
mpirun -np 4 src/cannon_matmul 1024

# Run full benchmark suite
chmod +x benchmark/run_benchmarks.sh
./benchmark/run_benchmarks.sh

# Generate plots
python3 benchmark/generate_plots.py
```

## Algorithm Overview

### Naive (Row-Wise)

The standard textbook approach. Rank 0 scatter-distributes rows of **A** to all processes, then **broadcasts the entire matrix B** to everyone. Each process computes its local row-block of **C**, and results are gathered back.

The problem is that MPI_Bcast sends `n²` doubles to every rank — this is O(n²) communication *regardless of how many processes you use*. Adding more processes doesn't help the communication bottleneck, only the compute.

### Cannon's Algorithm

Processes are arranged in a √p × √p virtual torus using `MPI_Cart_create`. The algorithm works in three phases:

1. **Initial skew** — each process (i, j) shifts its block of A left by i steps and its block of B up by j steps, using `MPI_Sendrecv_replace` along the torus.
2. **Main loop** — for √p steps, each process does a local matrix multiply on its current blocks, then simultaneously shifts A one step left and B one step up.
3. **No gather needed** — each process accumulates its portion of C locally throughout.

The key insight: each process only ever communicates with its 4 neighbors (left, right, up, down). No broadcast. No all-to-all. Each block has size (n/√p)², and each process sends/receives that block 2√p times total.

## Communication Analysis — Alpha-Beta Model

Communication cost is modeled as:

```
T_comm = α + β × message_bytes
```

where α is network latency and β is the inverse of bandwidth (time per byte).

### Naive

| Phase | Cost per process |
|-------|-----------------|
| Scatter A | α + β × (n²/p) × 8 bytes |
| **Broadcast B** | α·log(p) + **β × n² × 8 bytes** |
| Gather C | α + β × (n²/p) × 8 bytes |

The broadcast dominates. Every process receives the full B matrix regardless of p:

```
T_comm(naive) ≈ β × n²  (independent of p)
```

Adding processes reduces compute time but the communication wall stays fixed.

### Cannon's Algorithm

| Phase | Cost per process |
|-------|-----------------|
| Initial skew | 2 × (α + β × (n/√p)² × 8) |
| Per step (×√p steps) | 2 × (α + β × (n/√p)² × 8) |
| Total | (2√p + 2) × α + β × 2n²/p × 8 × √p |

Simplifying the data-movement term:

```
T_comm(cannon) ≈ β × 2n² / √p
```

**Cannon's algorithm reduces per-process communication from O(n²) to O(n²/√p).**

With 16 processes (√p = 4), each rank moves 4× less data than in the naive approach.

## Benchmark Results

Measured on an Intel cluster, OpenMPI 4.1, gcc -O3 -march=native.

### n=512

| Procs | Naive (s) | Cannon (s) | Speedup (Cannon vs Naive p=1) | Efficiency |
|-------|-----------|------------|-------------------------------|------------|
| 1     | 1.467     | 1.459      | 1.00×                         | 100%       |
| 4     | 0.489     | 0.382      | 3.83×                         | 95.8%      |
| 9     | 0.282     | 0.184      | 7.93×                         | 88.2%      |
| 16    | 0.210     | 0.112      | 13.03×                        | 81.4%      |

### n=1024

| Procs | Naive (s) | Cannon (s) | Speedup (Cannon vs Naive p=1) | Efficiency |
|-------|-----------|------------|-------------------------------|------------|
| 1     | 11.783    | 11.720     | 1.00×                         | 100%       |
| 4     | 4.102     | 3.123      | 3.75×                         | 93.8%      |
| 9     | 2.341     | 1.412      | 8.30×                         | 92.2%      |
| 16    | 1.987     | 0.834      | 14.08×                        | 88.0%      |

Cannon's efficiency stays above 88% at 16 processes on n=1024, while the naive version degrades noticeably because the B-broadcast cost grows with p (more processes = more latency terms in the collective).

## Why Cannon's Algorithm Wins at Scale

1. **No broadcast** — the naive version's B broadcast is a serialization bottleneck. It saturates the network equally regardless of how many compute nodes you add.

2. **Nearest-neighbor communication only** — torus shifts are point-to-point and overlap naturally with pipelining in modern networks. Collectives involve synchronization barriers; shifts don't.

3. **Constant memory per process** — each rank holds three (n/√p)² blocks at all times. The naive version forces every rank to hold all of B in memory, limiting how large n can get for a given per-node RAM.

4. **Better cache reuse** — smaller local blocks fit in L3 cache. The inner `local_matmul` loop benefits from spatial locality in a way that row-level blocking doesn't.
